## Context

### Problem Statement

The Resource System is a pure data registry — no simulation, no state machine, just definitions. The game needs a portable, data-driven format for resource definitions that game designers can edit without touching code, supports schema versioning (new fields, deprecation, backward compatibility), and provides O(1) runtime lookup by resource ID. All gameplay systems that handle items (Inventory, Production, Hunger, Trading, HUD, Building) reference this registry rather than hardcoding resource types.

### Constraints

- **Data-driven**: Designers must be able to add new resource types by editing a JSON file — no code changes required.
- **Schema versioning**: New fields can be added without breaking old saves (migration on load). Deprecated resources must remain loadable.
- **O(1) lookup**: Runtime access by resource ID must be dictionary/hash map — no linear search.
- **Fail-fast validation**: Invalid registry data must be rejected at load time, not silently ignored.
- **Godot 4.6**: Must use Godot 4.6 best practices for file I/O (`FileAccess.open()` returns `FileAccess` object or null; `store_*` methods return `bool` in 4.4+).
- **No engine-native Resource format**: The registry must be editable outside Godot (spreadsheet, config tools).

### Requirements

- JSON file format with version number and resources array
- Required fields: id, display_name, category, stack_limit, icon_path
- Optional fields: subcategory, weight, base_value, max_charge, description, tags
- Schema validation on load (reject invalid data, fail fast)
- O(1) id-keyed runtime lookup via dictionary
- Two-category model: Consumables / Production Goods
- Version-based migration for schema changes

## Decision

The Resource Data Registry will be implemented as:

1. **A JSON file** (`res://data/resources.json`) loaded at project startup by an Autoload singleton `ResourceRegistry`.
2. **A pure JSON file** (not Godot `.tres`) — editable outside Godot, version-control friendly, tool-agnostic.
3. **An Autoload singleton `ResourceRegistry`** (`extends Node`) that parses the JSON, validates the schema, caches entries in an `Dictionary[StringName, _ResourceDefinition]`, and exposes read-only lookup methods.
4. **Strict schema validation** with fail-fast: invalid data halts loading with a clear error message.
5. **Version-based migration**: `version` field in the JSON enables forward compatibility (new game loads old saves). Schema changes add optional fields only; required field additions trigger a version bump and migration.

### Why JSON (not Godot Resource files)?

Godot `.tres`/`.resource` files are engine-native but:
- Cannot be edited outside Godot
- Are harder to version-control (often binary-serialised)
- Cannot be generated by spreadsheet tools or config scripts
- Require engine dependency for design-time editing

JSON is:
- Universal format, editable in any text editor
- Diff-friendly for version control
- Can be generated from spreadsheets/config tools
- Supports explicit versioning
- Godot has built-in `JSON.parse()` with error reporting

### Why Autoload (not scene-instanced)?

The Resource Registry is pure data — no scene hierarchy needed. An Autoload provides:
- Always available from any system without dependency injection
- Loads at project start, before any scene
- Zero overhead (no scene management)
- Simplest pattern for a read-only data registry

### Key Interfaces

**Methods (called by other systems):**
```gdscript
enum ResourceCategory { CONSUMABLE, PRODUCTION_GOOD }

const _VALID_CATEGORY_STRINGS: Array[String] = ["consumable", "production_good"]

func load_from_file(path: String) -> bool
    # Opens JSON file, parses, validates, caches.
    # Returns false on any validation failure (fail-fast).
    # Must be called before any get_definition() calls.

func get_definition(id: StringName) -> _ResourceDefinition?
    # Returns the full resource definition struct, or null if not found.
    # O(1) lookup via internal Dictionary.

func is_valid_id(id: StringName) -> bool
    # Quick check: does this resource ID exist in the registry?
    # O(1) lookup.

func get_all_by_category(category: ResourceCategory) -> Array[_ResourceDefinition]
    # Returns all resources matching the given category.
    # O(n) filter — acceptable for startup-only queries.
    # Returns a fresh Array each call — callers must not mutate the returned reference.
```

**Internal Data Structure:**
```gdscript
class _ResourceDefinition:
    var id: StringName
    var display_name: String
    var category: ResourceCategory  # enum: CONSUMABLE, PRODUCTION_GOOD
    var stack_limit: int
    var icon_path: String
    var subcategory: String = ""
    var weight: float = 0.0
    var base_value: int = 0
    var max_charge: float = 100.0
    var description: String = ""
    var tags: Array[String] = []
    var deprecated: bool = false
```

**Load Order Contract:**
ResourceRegistry must be fully loaded (load_from_file succeeds) before any other system attempts to reference resources. This is enforced by WorldSaveManager's load order invariant: ResourceRegistry loads first.

### Internal Validation Rules
```gdscript
func _validate_resource(entry: Dictionary, index: int) -> Array[String]:
    var errors: Array[String] = []
    # Required field checks
    if not entry.has("id") or not entry["id"] is String:
        errors.append("Resource at index %d: missing or invalid 'id'" % index)
    if not entry.has("display_name") or not entry["display_name"] is String:
        errors.append("Resource at index %d: missing or invalid 'display_name'" % index)
    if not entry.has("category"):
        errors.append("Resource at index %d: missing 'category'" % index)
    elif entry["category"] not in _VALID_CATEGORY_STRINGS:
        errors.append("Resource at index %d: invalid category '%s'" % [index, entry["category"]])
    if not entry.has("stack_limit") or not (entry["stack_limit"] is int and entry["stack_limit"] >= 1):
        errors.append("Resource at index %d: missing or invalid 'stack_limit'" % index)
    if not entry.has("icon_path") or not entry["icon_path"] is String:
        errors.append("Resource at index %d: missing or invalid 'icon_path'" % index)
    # Cross-field validation
    if entry.has("max_charge") and entry["max_charge"] is float:
        if entry["max_charge"] <= 0.0:
            errors.append("Resource %s: max_charge must be > 0.0" % entry.get("id", "???"))
    return errors
```

## Alternatives Considered

### Alternative A: Godot Resource Files (.tres / .resource)

**Description**: Each resource is a `.tres` file (or instance of a `.resource` script). `ResourceLoader.load()` retrieves them at runtime.

**Pros**: Engine-native; type-safe via `class_name`; integrated with Godot's resource management; automatic deep-copy via `duplicate_deep()`.

**Cons**: Cannot be edited outside Godot; harder to version-control (diff noise); cannot be generated by spreadsheet tools or external scripts; no simple way to enumerate all resources at startup without filesystem traversal; Godot's resource format changes between minor versions (backward compatibility risk).

**Rejection Reason**: The project requires data-driven design — designers must be able to edit resource definitions in text editors, spreadsheets, or config tools without opening Godot. JSON is the universal format for this.

### Alternative B: Binary Custom Format

**Description**: Serialize resource definitions to a compact binary format with a custom header containing version number. Read/write via `FileAccess.get_*` / `FileAccess.store_*`.

**Pros**: Faster loading; smaller file size; harder to accidentally corrupt.

**Cons**: Cannot be edited manually or in spreadsheets; requires custom parser/writer code; debugging data issues is harder; no standard tooling.

**Rejection Reason**: The game's resource count for the Vertical Slice is small (~20-30 resources). File size and load time are irrelevant. JSON's human-editability and tool-agnosticism are far more valuable than binary performance for this scale.

### Alternative C: GDScript Array/Dictionary (Hardcoded)

**Description**: Resource definitions are inline `const` arrays/dictionaries in a GDScript file. `ResourceRegistry` reads them as a script constant.

**Pros**: No file I/O; type-checked at compile time; fastest possible lookup.

**Cons**: Not data-driven — adding a resource requires code changes; no schema versioning; no deprecation support; hard to version-control (diffs are code diffs, not data diffs).

**Rejection Reason**: Directly violates the GDD requirement that new resources can be added without code changes. The Resource System must be a data registry, not a hardcoded list.

## Consequences

### Positive
- **Data-driven**: Designers edit JSON, not code. New resources are one entry away.
- **O(1) lookup**: Dictionary-based cache provides instant access by resource ID.
- **Fail-fast validation**: Invalid data is caught at load, not during gameplay.
- **Version migration**: Old saves load via schema version check + default values for new fields.
- **Tool-agnostic**: JSON can be generated from spreadsheets, scripts, or any tool.
- **Godot-idiomatic**: FileAccess + JSON.parse() is a standard pattern, unchanged across versions.

### Negative
- **JSON parse overhead**: Every game start requires file I/O + parsing. For 30 resources, this is ~1-2ms — acceptable.
- **StringName memory**: Resource IDs cached as `StringName` persist for the entire session. For a small game this is fine.
- **No hot-reloading**: Changes to the JSON file require a game restart. Acceptable for Vertical Slice scope.
- **Autoload coupling**: ResourceRegistry as Autoload creates a global dependency. Mitigated by read-only nature — no state mutations, no coupling risk.

### Risks
- **Malformed JSON**: `JSON.parse()` returns an error code and a partial result on failure. Implementation must check the error code before accessing results. Mitigation: always check return value, display clear error message with file and line number.
- **Schema drift**: Designers add inconsistent data (e.g., duplicate IDs, wrong category values). Mitigation: design-time validation script (pre-commit) + runtime fail-fast.
- **Large resource count**: If the game grows to 500+ resources, `get_all_by_category()` O(n) filter becomes noticeable. Mitigation: maintain a secondary category-index dictionary at load time.

## Performance Implications
- **CPU**: `load_from_file()` runs once at startup (~1-2ms for 30 resources). Runtime lookups are O(1) dictionary access. `get_all_by_category()` is O(n) — acceptable for startup-only queries.
- **Memory**: ~30 `_ResourceDefinition` instances + Dictionary overhead. Negligible (< 1 MB).
- **Load Time**: Single JSON file read + parse. No streaming or async needed at Vertical Slice scale.
- **Network**: N/A (single-player only).

## Migration Plan

This is a new system — no migration from existing code. The file `res://data/resources.json` will be created with the initial resource set (wood, stone, fiber, berries, plank, tool, etc.) matching the resource-system.md GDD.

## Validation Criteria
- [ ] `load_from_file("res://data/resources.json")` returns true with valid file
- [ ] `load_from_file()` returns false with missing/malformed file
- [ ] `get_definition("wood")` returns wood's full definition including max_charge
- [ ] `get_definition("nonexistent")` returns null (no crash)
- [ ] `is_valid_id("wood")` returns true; `is_valid_id("???")` returns false
- [ ] `get_all_by_category(ResourceCategory.CONSUMABLE)` returns only consumables
- [ ] Invalid schema (missing required field) → load fails with error message
- [ ] Invalid max_charge (≤ 0.0) → load fails with error message
- [ ] Deprecated resources remain loadable but hidden from new creation
