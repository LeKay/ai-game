# ADR-0004: Grid Map Data Model and TileMapLayer Rendering

## Status

Accepted

## Date

2026-05-25

## Last Verified

## Decision Makers

Lead Programmer, Technical Director

## Summary

The game requires a 30×30 authoritative tile grid with three independently mutable data layers. We implement a `WorldGrid` Node (not Autoload) owning three `Array[Array]` data layers; `TileMapLayer` nodes are pure rendering targets that only receive `set_cell()` calls — they never act as a data source for gameplay queries.

## Context

### Problem Statement

The game requires a spatial data model for a 30×30 tile world with three independently mutable concerns: terrain (generated once, immutable after generation), resources (cleared by harvesting and building placement), and buildings (placed and removed by gameplay). These three concerns must be queried together (e.g. "can I place here?") but mutated independently. A rendering system must reflect the data state visually without owning the data.

### Current State

No grid system exists. This is a greenfield decision.

### Constraints

- Godot 4.6 deprecates `TileMap` (deprecated since 4.3); `TileMapLayer` is the required API
- `class_name GridMap` is a hard parser error in Godot 4.6 ("Class hides a native class") — the custom class must use a different name
- Grid must be queryable at O(1) per tile — no scan-based lookups in hot paths
- 60fps with up to 50 `get_tile_view()` calls per frame must complete in < 0.1ms total
- WorldGrid must not be an Autoload — it is a spatial data node, not a global service

### Requirements

- 30×30 grid, three layers: TerrainLayer (write-once after generation), ResourceLayer (mutable), BuildingLayer (mutable)
- O(1) read for all three layers per tile via array indexing
- Single placement validation gate checks all three layers atomically
- Procedural terrain generation with deterministic seed
- TileMapLayer nodes used for rendering only — no gameplay reads via `get_cell()`
- Coordinate conversion: tile ↔ world-space pixels

## Decision

`WorldGrid` is a `Node` (not Autoload) instantiated as a child of `MapRoot`. It owns three `Array[Array]` data layers and exposes a read/write API. `TileMapLayer` nodes are siblings in the `MapRoot` scene and receive one-way `set_cell()` calls after data mutations; they never serve as a data source.

### Architecture

```
MapRoot (Node2D)  y_sort_enabled = true
├── BackgroundLayer (TileMapLayer) — rendering only; always renders EMPTY (grass) for every tile
├── TerrainLayer (TileMapLayer)    — rendering only; non-EMPTY terrain features (trees, stone, etc.)
├── ResourceOverlay (TileMapLayer) — rendering only; resource indicator dots
├── BuildingSlots (TileMapLayer)   — rendering only
└── WorldGrid (Node)               — authoritative data; no rendering
      ├── _terrain: Array[Array]   — [x][y] → TileType (int), write-once
      ├── _resources: Array[Array] — [x][y] → ResourceTileData or null
      └── _buildings: Array[Array] — [x][y] → String (building_id) or null

Data flow: WorldGrid → TileMapLayer (set_cell only)
           TileMapLayer → WorldGrid: FORBIDDEN
```

### Key Interfaces

```gdscript
class_name WorldGrid extends Node

const GRID_SIZE: int = 30
const TILE_SIZE: int = 48

enum TileType { EMPTY, TREE, STONE, BERRY, GRASS, IMPASSABLE }
enum PlacementResult { SUCCESS, BLOCKED_BY_BOUNDS, BLOCKED_BY_IMPASSABLE,
                       BLOCKED_BY_BUILDING, BLOCKED_BY_RESOURCE_TILE }
enum DistanceMetric { MANHATTAN, EUCLIDEAN }

class ResourceTileData:
    var resource_id: StringName
    var clearable: bool

class TileView:
    var terrain: TileType
    var resource: Variant  # ResourceTileData or null
    var building_id: String

# Read API — all assert on out-of-bounds; O(1) array access
func get_terrain(tile: Vector2i) -> TileType
func get_resource(tile: Vector2i) -> Variant       # ResourceTileData or null
func get_building(tile: Vector2i) -> String        # "" if none
func get_tile_view(tile: Vector2i) -> TileView
func is_passable(tile: Vector2i) -> bool
func is_in_bounds(tile: Vector2i) -> bool          # safe pre-check, no assert

# Placement — single validation gate
func validate_placement(tile: Vector2i, building_type: String) -> PlacementResult
func place_building(tile: Vector2i, building_id: String) -> PlacementResult
func remove_building(tile: Vector2i) -> bool

# Resource mutation
func harvest_resource(tile: Vector2i, _amount: int) -> int  # 0 or 1; _amount reserved

# Coordinate conversion
func tile_to_world(tile: Vector2i) -> Vector2      # pixel center of tile
func world_to_tile(world_pos: Vector2) -> Vector2i # floor division

# Distance and spatial queries
func manhattan_dist(a: Vector2i, b: Vector2i) -> int
func euclidean_dist(a: Vector2i, b: Vector2i) -> float
func distance_between(a: Vector2i, b: Vector2i, metric: DistanceMetric) -> float
func get_tiles_in_radius(center: Vector2i, radius: int) -> Array[Vector2i]  # square box
func get_neighbors(tile: Vector2i, diagonals: bool = false) -> Array[Vector2i]
func find_nearest(tile: Vector2i, resource_id: StringName, max_radius: int) -> Variant
func find_tiles_by_predicate(predicate: Callable) -> Array[Vector2i]

# Serialization
func serialize() -> Dictionary
func deserialize(data: Dictionary) -> void
```

### Implementation Guidelines

1. **Class name**: Use `class_name WorldGrid`. Never use `class_name GridMap` — Godot 4.6 hard parser error.
2. **Array typing**: `_terrain` inner rows must be `Array[int]` for type safety. `_resources` and `_buildings` inner rows use untyped `Array` (hold null-or-object).
3. **Nullable returns**: `get_resource` returns `Variant` (not `ResourceTileData`) because GDScript 4 has no nullable type annotation. Callers must null-check.
4. **TerrainLayer immutability**: After `generate()` sets `_generation_done = true`, any future write attempt to `_terrain` must `assert(false, "TerrainLayer is immutable after generation")`.
5. **Placement atomicity**: `place_building` calls `validate_placement` internally as the single gate. No consuming system implements its own placement check.
6. **Rendering sync**: `MapRoot` owns both WorldGrid and TileMapLayer nodes. After `generate()`, MapRoot calls a batch `_sync_tilemap()` pass: `BackgroundLayer` receives EMPTY atlas coords for every tile; `TerrainLayer` receives terrain-specific atlas coords only for non-EMPTY tiles (transparent PNGs composite naturally over the background). On `place_building`, MapRoot clears the `ResourceOverlay` cell for the affected tile. WorldGrid never calls `set_cell()`.
7. **Forbidden**: `TileMapLayer.get_cell()` from any gameplay code. `TileMap` node anywhere in the project. `class_name GridMap`.
8. **`_amount` parameter on `harvest_resource`**: Reserved for future quantity systems; permanently ignored in this design (Anno-style resources have no quantity). Document this at every call site.
9. **Coordinate conversion**: `tile_to_world` returns pixel center (`Vector2(tile) * TILE_SIZE + Vector2(TILE_SIZE/2, TILE_SIZE/2)`). `world_to_tile` uses `floori()` for correct negative-value behaviour.
10. **`get_tiles_in_radius`**: Returns a **square bounding box**, not a circle. Callers needing circular proximity must post-filter with `euclidean_dist`. Document this on every call site.

## Alternatives Considered

### Alternative 1: TileMap as data source

- **Description**: Use `TileMap.get_cell()` to read terrain/resource state at runtime instead of maintaining separate arrays
- **Pros**: Fewer data structures to maintain; rendering and data in one place
- **Cons**: Deprecated since Godot 4.3; query speed is O(dictionary lookup) not O(1) array; couples rendering to gameplay logic; multi-layer queries require three separate node calls
- **Rejection Reason**: Deprecated API; architectural coupling between rendering and data

### Alternative 2: Autoload singleton for WorldGrid

- **Description**: Register WorldGrid as an Autoload so any system can call `WorldGrid.get_terrain()` globally
- **Pros**: Easy access from any script without dependency injection
- **Cons**: Violates ADR-0006 load-order invariant (WorldGrid must be instantiated after ResourceRegistry, which Autoload cannot guarantee); makes the grid a global service rather than a scene-scoped node; prevents multiple grid instances in tests
- **Rejection Reason**: ADR-0006 load-order; testability; scene-scoped ownership model is correct

### Alternative 3: Dictionary-keyed storage

- **Description**: Store tile data in `Dictionary[Vector2i, TileData]` instead of `Array[Array]`
- **Pros**: Sparser storage; natural null-means-empty semantics
- **Cons**: O(hash) access vs O(1) array indexing; 900-element fixed grid makes sparse storage pointless; hashing Vector2i adds per-query overhead
- **Rejection Reason**: Performance; fixed grid size makes arrays strictly better

## Consequences

### Positive

- O(1) array indexing for all tile queries
- Clean separation: WorldGrid owns data, TileMapLayer owns rendering
- Single placement gate prevents inconsistent state across systems
- Deterministic generation enables reliable testing and replay
- Testable without a scene tree (unit tests instantiate WorldGrid directly)

### Negative

- `get_resource` returns `Variant` instead of a typed nullable — callers must remember to null-check
- Rendering sync (`_sync_tilemap`) must be called explicitly by MapRoot after mutations — not automatic
- `class_name WorldGrid` diverges from the file name `grid_map.gd` — file should be renamed to `world_grid.gd` in a future cleanup

### Neutral

- `_generation_done` flag adds a write-once invariant that requires explicit enforcement at each terrain-write path
